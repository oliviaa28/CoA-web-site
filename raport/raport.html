1. Introducere

1.1 Scop

Acest document reprezinta Specificatia de Cerinte Software pentru aplicatia web CoA - Crisis Containment Service, realizata in cadrul cursului de Tehnologii Web, anul universitar 2025-2026. Documentul descrie cerintele functionale si non-functionale ale platformei, arhitectura adoptata si principalele decizii de implementare.

1.2 Scopul produsului

CoA este o platforma web pentru gestionarea situatiilor de urgenta (cutremure, inundatii, incendii) de catre autoritatile competente. Aplicatia ofera un panou de administrare complet, vizualizare geografica a evenimentelor si adaposturilor, emiterea de alerte in format CAP (Common Alerting Protocol), import de date istorice din seturi de date publice si export in formate deschise. Totodata, pune la dispozitia cetatenilor o interfata publica de informare, fara autentificare.

1.3 Referinte

  • OASIS CAP v1.2 - docs.oasis-open.org
  • Kaggle - Global Earthquake Events Dataset - kaggle.com
  • Kaggle - Cutremure Romania 2012-2024 - kaggle.com
  • W3C Scholarly HTML - w3c.github.io/scholarly-html
  • IEEE SRS Template - github.com
  • Open-Meteo Weather API - open-meteo.com
  • Leaflet.js - leafletjs.com

2. Descriere generala

2.1 Perspectiva produsului

CoA este o aplicatie web independenta, construita fara framework-uri externe la nivel de client sau server. Arhitectura urmeaza pattern-ul MVC cu Page Controller: fiecare pagina are propriul controller PHP, cu modele separate pentru accesul la date si view-uri HTML pentru prezentare.

Comunicarea dintre frontend si backend se realizeaza asincron prin Fetch API, serverul expunand endpoint-uri JSON pentru toate operatiile CRUD. Aceasta plaseaza aplicatia in categoria arhitecturii bazate pe servicii web.

2.2 Functiile principale

  • Autentificare si gestionarea sesiunilor pentru administratori
  • CRUD complet pentru evenimente (cutremure, inundatii, incendii)
  • CRUD complet pentru adaposturi cu localizare geografica
  • Creare si export de alerte in format CAP (XML)
  • Harta interactiva cu Leaflet.js pentru vizualizarea evenimentelor si adaposturilor
  • Import si export de date in format JSON si CSV
  • Import de date istorice despre cutremure din seturi de date Kaggle
  • Gestionarea conturilor de administrator
  • Interfata publica de informare pentru cetateni

2.3 Utilizatori

Aplicatia are doua tipuri de utilizatori. Administratorul este o autoritate acreditata care are acces complet la panoul de administrare si poate gestiona toate entitatile platformei. Publicul reprezinta orice persoana care acceseaza platforma pentru informatii, fara autentificare si fara posibilitatea de a modifica date.

2.4 Mediul de operare

Aplicatia ruleaza pe un server Apache cu suport PHP, folosind MySQL ca baza de date relationala, accesat prin PDO. In dezvoltare am folosit XAMPP local. Interfata functioneaza in orice browser modern pe desktop, tableta sau telefon.

2.5 Constrangeri de implementare

  • Fara framework-uri frontend - JavaScript fara librarii externe
  • Fara framework-uri backend - PHP cu MVC implementat manual
  • HTML si CSS valide conform specificatiilor W3C
  • Toate interogarile SQL folosesc PDO cu prepared statements
  • Comunicarea asincrona prin Fetch API pentru operatii CRUD
  • Cod gestionat prin Git cu branch-uri separate per functionalitate

2.6 Dependente

  • Leaflet.js - biblioteca open-source pentru harti interactive
  • Open-Meteo API - date meteo in timp real, acces public
  • ip-api.com - geolocalizare dupa IP, acces public
  • ipify.org - detectarea IP-ului public
  • Date Kaggle - importate o singura data prin scripturi PHP dedicate

3. Interfete externe

3.1 Interfata utilizator

Aplicatia are doua seturi de interfete distincte.

Interfata publica

  • events_public.php - lista evenimentelor cu filtrare si harta
  • details_public.php - detalii eveniment individual
  • shelter_public.php - lista adaposturilor cu localizare

Panoul de administrare

  • login.html - autentificare
  • dashboard.php - statistici generale
  • events.php - gestionare evenimente
  • shelters.php - gestionare adaposturi
  • alerts.php - gestionare alerte si export CAP
  • users.php - gestionare administratori
  • import-export.php - import si export fisiere

Toate paginile sunt responsive. Notificarile de succes si eroare sunt afisate prin componente toast generate dinamic in JavaScript.

3.2 Interfete software

3.3 Interfete de comunicare

  • HTTP - protocolul de transfer intre browser si serverul Apache
  • Fetch API - comunicare asincrona pentru CRUD, date serializate JSON
  • CAP over XML - alertele exportate ca XML conform OASIS CAP 1.2
  • multipart/form-data - upload fisiere la import
  • Sesiuni PHP - autentificarea mentinuta prin cookie PHPSESSID

4. Functionalitati

4.1 Autentificare si sesiuni

Sistemul de autentificare este bazat pe sesiuni PHP. Utilizatorul completeaza emailul si parola in formularul de login, browserul trimite un request POST, serverul verifica credentialele si creaza sesiunea daca sunt corecte.

  • Parolele sunt stocate exclusiv ca hash prin password_hash()
  • Verificarea parolei se face cu password_verify()
  • Datele sesiunii sunt stocate server-side, clientul primeste doar cookie-ul PHPSESSID
  • Toate paginile admin verifica sesiunea activa prin AuthController::requireAuth()
  • La logout, sesiunea este distrusa complet cu session_destroy()

4.2 Gestionare evenimente

Functionalitate centrala a platformei - CRUD pe trei tipuri de incidente: cutremur, inundatie, incendiu.

  • Fiecare tip de eveniment are tabel propriu in baza de date
  • Toate evenimentele au coordonate GPS pentru afisare pe harta
  • Filtrarea functioneaza combinat dupa tip, status si an
  • Un singur modal cu schimbare de mod (adaugare vs editare) gestionat prin JavaScript
  • Datele Kaggle sunt importate prin scripturi dedicate (import_kaggle.php, import_ro.php)
  • Toate operatiile CRUD se realizeaza asincron prin Fetch API

4.3 Gestionare adaposturi

CRUD pe adaposturi cu informatii despre capacitate si localizare geografica.

  • Fiecare adapost are: nume, adresa, tip, capacitate totala, locuri disponibile, coordonate GPS, descriere
  • Adaposturile sunt vizibile public fara autentificare
  • Localizarea geografica permite afisarea pe harta interactiva

4.4 Alerte CAP

Creare si export de alerte de urgenta conform standardului international OASIS CAP 1.2.

  • O alerta contine: tip incident, headline, descriere, instructiuni, severitate, urgenta, certitudine, zona
  • Exportul genereaza un document XML well-formed cu namespace urn:oasis:names:tc:emergency:cap:1.2
  • Documentul include elementele obligatorii: identifier, sender, sent, status, msgType, scope si blocul info
  • Tipul incidentului se mapeaza la categoriile CAP: cutremur - Geo, inundatie - Met, incendiu - Fire
  • Data este formatata conform ISO 8601 cu timezone
  • XML-ul este generat prin DOMDocument PHP, garantand well-formedness

4.5 Harta interactiva

Vizualizare geografica a evenimentelor si adaposturilor prin Leaflet.js.

  • Harta afiseaza markere pentru evenimentele active, diferentiate vizual dupa tip
  • Click pe marker deschide un popup cu informatii si link spre pagina de detalii
  • Datele pentru harta sunt preluate asincron prin Fetch API
  • Harta este responsiva si functioneaza pe dispozitive mobile

4.6 Import si export date

Transfer de date in formate deschise.

  • Export JSON cu indentare si suport unicode, descarcat direct din browser
  • Export CSV cu header pe prima linie, generat prin fputcsv() direct in output stream
  • Import JSON prin json_decode() aplicat pe fisierul uploadat
  • Import CSV cu citire linie cu linie si mapare chei-valori prin array_combine()
  • Validare structurala a fisierelor inainte de insertie in baza de date
  • Export alerte in format CAP (XML) prin DOMDocument

4.7 Gestionare utilizatori

CRUD pe conturile de administrator ale platformei.

  • La creare, parola este hashata cu password_hash()
  • La editare, parola se re-hasheaza doar daca a fost introdusa una noua
  • Stergerea unui utilizator este ireversibila

4.8 Interfata publica

Acces public la informatii despre urgente, fara autentificare.

  • Lista evenimentelor cu filtrare si harta, vizibila oricui
  • Pagina de detalii per eveniment cu statistici specifice tipului
  • Lista adaposturilor cu localizare si capacitate disponibila
  • Nicio operatie de scriere nu este accesibila fara autentificare

5. Cerinte non-functionale

5.1 Securitate

  • SQL Injection - toate interogarile folosesc PDO cu prepared statements
  • XSS - datele dinamice inserate in DOM folosesc textContent in loc de innerHTML; server-side se foloseste htmlspecialchars()
  • Parole - stocate exclusiv ca hash, verificate cu password_verify()
  • Sesiuni - date stocate server-side, clientul detine doar identificatorul de sesiune

5.2 Responsivitate

Toate paginile includ meta viewport si folosesc CSS Flexbox si Grid pentru adaptare la rezolutii diferite. Sidebar-ul admin se adapteaza pe ecrane mici.

5.3 Calitate cod

  • Arhitectura MVC cu separare clara Model, View, Controller
  • Pattern Table Data Gateway pentru accesul la date
  • Cod comentat
  • Gestionare prin Git cu branch-uri per functionalitate
  • HTML validat prin W3C Markup Validation Service

5.4 Portabilitate

Include-urile PHP folosesc cai absolute, iar configuratia bazei de date este centralizata in config/database.php.
